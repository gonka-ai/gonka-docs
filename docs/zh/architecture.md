# 架构

## 推理请求在 Gonka 网络中的流转

下图及描述展示了推理请求如何在 Gonka 网络中流动。Gonka 是一个去中心化网络，旨在平衡性能与密码学保障。  
Gonka 仅记录用于推理验证的交易和验证凭证，实际计算在链下进行。

推理请求在独立的“主机”（Host，即所有网络参与者，而非中心化调度器）之间流转。该系统为去中心化架构，没有单一节点负责将推理请求分发至网络中的节点。实践中，每个 Host 会部署至少两个节点：

- **网络节点**：负责通信，包含：
 
 
    - **链节点（Chain Node）**：连接区块链
 
 
    - **API 节点（API Node）**：管理用户请求
- **一个或多个机器学习节点（ML Node）**：执行大语言模型（LLM）推理任务（ML 节点可部署在多个服务器上）。

以下是推理请求在 Gonka 网络中流转的步骤（如上图所示）：

- **步骤 1**：客户端（开发者）从活跃参与者（Host）列表中随机选择一个节点。该随机选定的 Host 作为“传输代理”（Transfer Agent, TA），其 API 节点接收推理请求。任何 Host 都可充当验证者（Validator）、传输代理（TA）和执行者（Executor）——这些并非预定义或链上角色，而是处理请求时动态承担的操作职能。
- **步骤 2**：TA 从其他所有活跃 Host 中随机选择一个作为“执行者”（Executor），并将推理输入传递至该 Executor 的 API 节点。同时，TA 的链节点将推理输入记录上链。需要注意的是，链上记录不会阻塞 LLM 计算过程，而是与 Executor 的计算并行进行。
- **步骤 3**：Executor 的 API 节点将请求转发至其某个 ML 节点，该节点立即开始执行推理。
- **步骤 4**：计算完成后，Executor 的 ML 节点将推理结果返回给其 API 节点。
- **步骤 5**：Executor 的 API 节点将结果回传至 TA 的 API 节点，同时其链节点将验证凭证记录上链。
- **步骤 6**：TA 的 API 节点将最终结果返回给客户端（开发者），同时 TA 的链节点也将结果上链。需要注意的是，这些链上记录虽会影响整体网络带宽，但不会增加单次推理计算的延迟。

### 性能与验证

链上记录不会影响推理计算的启动时间，也不会延迟客户端获取最终结果的时间。推理是否诚实执行的验证在后续并行进行，与其他推理任务同时发生。如果发现 Executor 存在作弊行为，其将失去整个周期的奖励，客户端会收到通知并获得退款。  
需注意，该图仅从高层展示推理流程，未体现网络的全部复杂性。例如，未显示 Host 之间链节点的直接通信，也未包含桥接器及其他与此抽象层级无关的内部组件。

## 计算证明（Proof of Compute, PoC）时间线

**计算证明**（PoC）是一种新型共识机制，继承了传统工作量证明（PoW）的优点，即权重、工作量和奖励与计算能力对齐（不同于权益证明 PoS，其权重与质押代币数量挂钩）。与 PoW 不同的是，PoC 将“证明”部分（称为“冲刺”Sprint）集中在短暂且有限的时间窗口内完成，其余时间则用于执行有用工作（如大语言模型推理）。

Gonka 以“周期”（epoch）为单位运行，每个周期持续 17280 个区块（约 24 小时）。

每个周期遵循严格的流程，将 Sprint 执行、Host 活动和奖励结算整合为一个连贯的整体。

所有 Host 同时开始 Sprint，确保公平性（类似于赛跑的起跑枪，任何 Host 都不能提前开始）。这种同步启动通过一个无法预测或提前操控的随机种子来协调。所有具有投票权的 Host 共同参与生成该种子，防止少数 Host 提前计算或获得不公平优势。

Sprint 被设计为极短的时间窗口，将计算资源集中在高效、紧凑的区间内。Sprint 按照区块链出块节奏定期、精确地发生，仅占整个周期的一小部分。通过最小化 Sprint 持续时间，使周期的大部分时间可用于执行有意义的工作，如 LLM 推理与训练。这种可预测的节奏使 Sprint 能自然融入去中心化 AI 网络的整体运行，而不干扰生产性任务。

一个周期以自动申领第 N 轮奖励结束。新的计算证明阶段（Sprint）随即启动，用于确定下一周期的 Host 权重。Sprint 完成并分配权重后，即标志着新周期的开始。

在整个周期中，Host 持续执行和验证推理任务。

如果因某些原因未能及时申领第 N 轮奖励，每个 Host 的 API 节点将在第 N+1 轮自动提交一笔奖励申领交易（针对第 N 轮），使用第 N 轮开始时签名的种子。该请求每 30 分钟重试一次，直至成功。重要的是，Host 必须在此有限窗口期内保持在线并通过所有验证检查，否则链上无法完成申领，奖励将无法获得。未申领的奖励（第 N 轮）将在第 N+2 轮开始时被永久销毁。

在申领过程中，链会验证 Host 是否完成了该周期的所有必要工作。协议还允许在此窗口期内提交逾期的推理验证凭证，为 Host 提供最后一次机会，以完成任何待处理的验证任务，然后再最终确定奖励。
