# 仪表板维护者备忘录 — Gonka v0.2.14

本次升级未移除或重塑任何现有查询端点。如果您的仪表板从链上读取值并显示，它将继续正常工作。您需要理解一个语义变更（委托），以及三个小注意事项。

## 委托：权重 vs 奖励

**之前（v0.2.13）：** 当A委托给B时，链在存储权重之前会将A的部分权重物理转移给B。您从链上读取的`weight`已经包含了委托影响。奖励会自动根据调整后的权重计算。

**现在（v0.2.14）：** 链存储每个人的*自身*权重，不受委托影响。委托仅在奖励结算时内部应用，作用于一个从未作为链字段暴露的权重副本。（拒绝/不参与惩罚也变为仅影响奖励，不再减少存储的权重。）

这对您的仪表板意味着什么，取决于其功能：

- **您显示从链上读取的`weight`**（`epoch_group_data` → `validation_weights[].weight`，或`get_participant_current_stats`）：继续完全按原样操作。仍然正确。只需知道该数值不再包含委托影响——委托出去的参与者仍显示其完整权重。
- **您自行重新计算权重并在其上应用委托：** 停止应用委托。链不再这样做。
- **您估算预期奖励为“参与者权重 ÷ 总权重 × 期池”：** 只要存在委托、拒绝惩罚或排除，此方法现在就是错误的。请改用新端点：

`GET /chain-api/productscience/inference/inference/estimate_bitcoin_reward/{participant}`

它复现了链在结算时使用的相同计算逻辑——包含委托转移、惩罚和排除——并返回估算的奖励和工作币。该估算是针对当前期的：其输入在期形成时（PoC验证结束时）快照，并在每期过渡时刷新，因此该端点在整个期中均有效。NotFound响应表示参与者不在当前期的活跃集合中。

如果您想显示委托本身：谁委托给谁在`.../poc_delegation/{participant}`，份额参数在`params`（`delegation_params.delegation_share`）。

## 注意：奖励接收地址覆盖

参与者现在可以注册一个不同的地址来接收其领取的奖励。其身份、权重或统计数据均无变化——仅改变代币的接收地址。

这不构成兼容性破坏。仅当您的仪表板声称或假设“参与者X的奖励始终发送到地址X”时才重要——这不再有保证。来自`epoch_performance_summary` / `settle_amount`的每参与者奖励金额仍按参与者键值正确保留。如果您想显示覆盖地址，它位于`.../list_claim_recipients/{participant}`。

## 注意：一次性供应量下调

在升级高度，链将燃烧累积的费用收集器余额（金额在运行时确定，因此直到发生前未知），并将铸币通胀设为零。如果您绘制或估算总供应量，请预期在升级区块出现一次性下降，之后通胀线保持平坦。常规的`/cosmos/bank/v1beta1/supply/by_denom?denom=ngonka`查询仍可正常工作。

## 注意：已弃用的推断端点

在dapi公共API上，`/v1/chat/completions`、`/v1/completions`和`/v1/devshard`下的所有内容现在返回HTTP 410 Gone（推断现在由版本化的devshard运行时提供，例如`/devshard/v3`）。仅当您曾将这些端点用作主机存活检查时才相关。`/v1/epochs/...`、参与者、模型、定价、状态——均未受影响。
