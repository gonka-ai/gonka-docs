# 奖励机制

!!! note 
    本文档仅供参阅。最终参数和技术细节可能发生变化，因此在依赖本文档前，请务必通过链上最新状态及相关 GitHub 仓库进行核实。

## 代币经济概览

Core 激励计划（6.8 亿枚 gonka (GNK) 代币，占总供应量的 68%）旨在奖励为去中心化网络贡献算力的节点（Hosts）。在类似比特币的奖励体系下，每个 epoch 固定量的 GNK 被铸造，并根据各节点的“算力证明”（Proof of Compute, PoC）权重按比例分配。随着网络扩展，更多 GPU 竞争相同的固定奖励，每台 GPU 获得的 GNK 逐渐减少，从而引入类似比特币挖矿的稀缺性价值机制。节点的长期收益来源于推理服务需求的增长、固定发行量带来的稀缺性，以及随着挖矿竞争加剧而上升的 GNK 市场价值。奖励计划遵循可预测的指数衰减模式，每 epoch 奖励逐步减少，大约每 4 年减半一次。

[代币经济完整版文档](https://gonka.ai/tokenomics.pdf)

## Gonka 网络的奖励系统

Gonka 网络采用双轨奖励机制，旨在平衡长期稀缺性激励与即时经济效用。节点通过两种互补的奖励渠道获得报酬：

- 协议层发行奖励（比特币风格）
- 基于实际推理需求的工作奖励

### 比特币风格奖励系统

Gonka 网络引入了受比特币启发的固定奖励机制，每个 epoch 铸造预定数量的奖励代币，并根据各节点的“算力证明”（PoC）权重进行分配。

该模型的关键特征包括：

- **每 epoch 铸造奖励代币**：每个 epoch 铸造基础数量为 323,000 GNK（通过治理设定，可能变动）的“奖励代币”（Reward Coins），并分发给所有活跃节点。该数值按指数衰减（每 epoch 衰减率 -0.000475，相当于约每 1,460 个 epoch 或 4 年减半一次），计算公式为：`current_epoch_reward = initial_reward × exp(decay_rate × epochs_since_genesis)`
- **稀缺性驱动价值**：随着更多 GPU 加入网络，每台 GPU 每 epoch 获得的 GNK 减少。单位 GNK 的挖矿成本上升，从而创造内在价值，并形成类似比特币的正向反馈循环：`More Hosts → fewer GNK per GPU → higher scarcity → potential price support and growth`

### 基于工作的奖励

与此同时，Gonka 网络还会根据实际经济活动分发“工作代币”（Work Coins）：

- 开发者为实际的推理和计算任务支付费用
- 这些费用将根据节点实际完成的计算量按比例分发给各节点

## 奖励释放机制（Vesting）

Gonka 网络分发的所有奖励，包括每 epoch 铸造的奖励代币和基于使用量的工作代币，均受协议强制执行的确定性释放机制约束，由专用的释放系统管理：

- **按节点独立记账**：网络为每个节点维护独立的释放计划，按日记录其奖励，确保分配的公平性和可预测性。所有节点遵循相同的释放规则和参数。
- **确定性奖励分配**：当节点获得新奖励时，系统将总额平均分配到整个释放周期内。任何因四舍五入产生的微小余额将加至第一天释放，确保无代币因精度损失而丢失。
- **自动管理**：系统自动处理所有释放操作，无需节点干预，既保持高效性，又避免系统臃肿。
- **每日释放**：每天，每个节点最旧的一笔待释放奖励将自动转入其可用余额，形成持续稳定的解锁流。

截至 2026 年 3 月，释放周期设定为 180 个 epoch（约 180 天）。有关最新释放计划，请始终参考当前链上实现。  
任何链下描述可能因协议升级、参数变更或治理决策而过时。

该机制将节点的奖励与网络的长期稳定和发展相绑定，防止短期投机行为，确保可靠节点获得可预期的收入流。

节点可查询：

- 待释放代币的总数量
```

http://<ip>:<public_http_port>/chain-api/productscience/inference/streamvesting/total_vesting/<Host_address>
```

其释放计划的详细分解（未来解锁的时间表数组）。
```

http://<ip>:<public_http_port>/chain-api/productscience/inference/streamvesting/vesting_schedule/<Host_address>
```

## 奖励申领流程
Gonka 链仅在第 N+1 轮次中发放第 N 轮次所获得的奖励。奖励无法提前或延后申领。

主机的 API 节点会在第 N+1 轮次期间，使用在第 N 轮次开始时记录签名的种子（seed）自动执行奖励申领操作。若申领失败，节点将每 30 分钟重试一次。

然而，申领操作要求节点在此有限的时间窗口内保持在线，并成功通过所有验证步骤，直至奖励成功申领。如果无法完成验证，链将无法最终确认奖励，奖励将保持未申领状态。

!!! note "关键运行提示"

    奖励申领操作**仅由主机的 API 节点**执行。
    
    如果在第 N+1 轮次期间，API 节点被停止、重启或处于不可用状态，
    则第 N 轮次的奖励可能会永久丢失，即使：
   
 
 
   - 已成功完成计算证明（PoC）
 
 
   - 推理请求已正确响应
 
 
   - ML 节点始终保持在线
    
    仅靠 ML 节点无法申领奖励。为确保奖励被安全申领，API 节点必须保持在线并同步状态，直至前一轮次的奖励被确认已成功申领。

在奖励申领过程中，链需验证主机已完成该轮次所有必要的工作。同时，链允许主机在申领窗口期内提交逾期的推理验证。这些检查要求保留所有推理输出及其元数据。

## 种子承诺与验证
种子是由每个网络节点在新一轮次开始前独立生成的一个秘密值。在轮次开始前，网络节点通过向链提交该种子的数字签名，完成对种子的承诺。  
在轮次结束时，为了获得奖励，节点必须公开其种子本身。链随后会验证公开的种子是否与先前记录的签名匹配。

示例结构：
```

{
  "seed": 5839402176541298043, // the seed itself, revealed during the claim 
  "epoch_index": 101,
  "signature": "a5910d9b4156e403494a0ba148e6fb67e101ab2a7b7cf54eb15fa5bb5a46a2fdb1435a14954221008d294987075b75d247f26fa7d273720478f0ffb3189132f6", // the seed signature, committed beforehand
  "claimed": false
}
```

该机制确保每个主机真正随机选择推理请求进行验证，且未对选择过程进行操控。同时，其他主机无法推断该节点如何执行随机选择，从而防止对抗性适应。

## 奖励未申领的原因

尽管申领流程是自动的，但并非无条件执行。

在第 N+1 轮期间，若发生以下情况，奖励可能保持未申领状态：

- API 节点离线或正在重启
- 节点无法提供正确的种子（终端性故障）
- 所需的推理-验证数据无法验证（终端性故障）
- 网络不稳定导致无法完成最终确认（可重试故障）

区块链不会在申领窗口期后保留推理数据，也无法处理逾期申领。  
此前各轮次中未申领的奖励将被永久销毁。

## 手动奖励申领

手动申领功能适用于以下操作人员：

- 正在停用或退役节点
- 在修复了导致第 N+1 轮期间自动申领失败的问题后，需手动触发申领
- 希望强制立即发起申领，而不愿等待自动重试周期

!!! note "运维说明"
    在第 N+1 轮期间短暂离线无需手动申领。  
    系统会在整个第 N+1 轮期间每约 30 分钟自动重试申领。  
    仅当节点无法及时恢复，或操作员希望在下次自动重试前立即尝试时，才需要手动申领。

    **手动申领无法恢复已过期轮次的奖励。**

    若 API 节点在第 N+1 轮期间离线且申领窗口已关闭，则第 N 轮的奖励将永久不可恢复。

    手动申领仅对上一个轮次有效，且前提是本地仍保存有正确的种子。

    手动奖励申领：
```

curl -X POST http://localhost:9200/admin/v1/claim-reward/recover \
    -H "Content-Type: application/json" \
    -d '{"force_claim": true, "epoch_index": 106}'
```

此端点仅在有效的申领周期（N+1）内有效。申领窗口关闭后，过期的奖励将无法再被领取。

## 申领验证流程

申领奖励依赖于节点在推理验证中的参与度。为确保随机性，每个主机（Host）需在周期开始时提交其将用于选择待验证推理任务的随机种子（random seed）的签名。

在执行申领操作时，节点需提供其在周期开始时已将签名记录在链上的随机种子。

区块链会验证所有用于验证的推理抽样事件是否确实由该种子生成。

为了完成此验证，系统需要访问所有的推理数据。由于存储成本较高，这些数据仅保留一个周期。

您可通过以下 API 查询推理性能相关信息：
```

GET /chain-api/productscience/inference/inference/epoch_performance_summary/<epochId>/<HostAddress>
```

示例
```

export NODE_URL=http://46.4.101.189:8000/
curl $NODE_URL/chain-api/productscience/inference/inference/epoch_performance_summary/84/gonka1q0a3u8s2azydlu2s2902qxwfz8d7mkze4lf45f | jq
```

返回指定纪元（epoch）和主机（Host）的性能摘要。

## 奖励可能被没收的条件

在以下情况下，主机可能失去获得奖励的资格：

**
1. 在奖励申领完成前关闭节点**

如果主机的 API 节点在奖励成功申领之前被停止、重启或变得不可用，区块链将无法完成申领流程。

**
2. 过多未完成的推理请求**

在整个纪元期间，系统会持续评估推理的正确性。系统跟踪两种独立的故障模式：

- 过多无效推理（Invalid Inferences）
- 过多未完成推理（Missed Inferences，未完成率超过 10%，小样本情况下的统计误差已作调整；详见代码实现）

这两个指标在整个纪元期间持续采样。如果任一指标在任何时刻超过协议定义的故障阈值，主机将立即面临以下后果：

- 从活跃主机列表中移除
- 抵押品被罚没（Collateral slashing）
- 没收当前纪元的所有奖励

该纪元内主机所完成的所有工作将不予补偿，所有已贡献的推理请求均视为无报酬。具体参数由协议层定义，并可能通过链上治理机制进行更新。主机应始终参考当前链上实现以获取权威行为说明。

**
3. 提前关闭或管理不当**

在证明计算能力（Proof of Capacity, PoC）完成前过早关闭节点，或在奖励申领最终确认前关闭，将导致奖励丧失。  
这些规则由协议层强制执行，不可逆转。

**
4. 确认型（随机）PoC 失败**

未能通过确认型 PoC 将对主机的收益产生直接且重大的影响。由于确认型 PoC 的存在是为了确保主机持续保有其声明的计算能力，一次失败将被视为该主机不再提供其声明 PoC 权重所依赖的算力的证据。因此，协议将施加严格的经济惩罚，包括：

- 没收当前纪元的所有奖励
- 通过按比例重新分配，将奖励隐性转移给其他符合条件的主机
- 抵押品被罚没
- 作废此前保留的 PoC 权重（该权重原本用于奖励计算）
