# 奖励机制

!!! note
    本文档仅作参考之用。最终参数和技术细节可能发生变化，因此在依赖本文档前，请务必核实链上最新状态及相关 GitHub 仓库。

## 代币经济概览

Core 激励部分（6.8 亿枚 gonka (GNK) 代币，占总供应量的 68%）旨在奖励为去中心化网络贡献算力的节点（Hosts）。在类似比特币的奖励系统下，每个纪元（epoch）会固定铸造一定数量的 GNK，并根据各节点的“算力证明”（Proof of Compute, PoC）权重按比例分配。随着网络扩张，更多 GPU 参与竞争相同的固定奖励，每台 GPU 获得的 GNK 数量将减少，从而引入类似比特币挖矿的稀缺性价值机制。节点的长期收益来源于推理服务需求的增长、固定发行量带来的稀缺性，以及随着挖矿竞争加剧而上升的 GNK 市场价值。奖励计划遵循可预测的指数衰减模式，纪元奖励逐步降低，大约每 4 年减半一次。

[代币经济完整版文档](https://gonka.ai/tokenomics.pdf)

## Gonka 网络的奖励系统

Gonka 网络采用双轨奖励机制，旨在平衡长期稀缺性激励与即时经济实用性。节点通过两种互补的奖励渠道获得报酬：

- 协议层发行奖励（比特币风格）
- 基于实际推理需求的工作奖励

### 比特币风格奖励系统

Gonka 网络引入了受比特币启发的固定奖励机制，每个纪元铸造预定数量的奖励代币，并根据各节点的“算力证明”（PoC）权重进行分配。

该模型的关键特征包括：

- **纪元铸造奖励代币**：每个纪元基础铸造 323,000 GNK（由治理设定，可能调整）作为“奖励代币”，并分发给所有活跃节点。该数值将按指数衰减（每纪元 -0.000475，相当于约每 1,460 个纪元或 4 年减半一次），计算公式为：  
  `当前纪元奖励 = 初始奖励 × exp(衰减率 × 自创世以来的纪元数)`
- **稀缺性驱动价值**：随着更多 GPU 加入网络，每台 GPU 每纪元获得的 GNK 减少。单位 GNK 的挖矿成本上升，从而产生内在价值，并形成类似比特币的正向反馈循环：  
  `更多节点 → 每 GPU 获得更少 GNK → 稀缺性上升 → 潜在价格支撑与增长`

### 基于工作的奖励

与此同时，Gonka 网络还会根据实际经济活动分发“工作代币”（Work Coins）：

- 开发者为实际的推理和计算任务支付费用
- 这些费用将根据节点实际完成的计算量按比例分发给各节点

## 奖励释放机制（Vesting）

Gonka 网络分发的所有奖励，包括纪元铸造的奖励代币和基于使用量的工作代币，均受协议强制执行的确定性释放机制约束，由专用的释放系统管理：

- **按节点独立记账**：网络为每个节点维护独立的释放计划，按日追踪其奖励，确保分配公平且可预测。所有节点遵循相同的释放规则和参数。
- **确定性奖励分配**：当节点获得新奖励时，系统会将总额平均分配至整个释放周期。任何因四舍五入产生的微小分数金额将加至第一天，确保无代币因精度问题丢失。
- **自动管理**：系统自动处理所有释放操作，无需节点干预，既提升效率又防止系统臃肿。
- **每日释放**：每天，每个节点最旧的一笔待释放奖励将自动转入其可用余额，形成稳定的代币解锁流。

截至 2026 年 3 月，释放周期设定为 180 个纪元（约 180 天）。有关最新的释放计划，请始终参考当前链上实现。  
任何链下描述可能因协议升级、参数变更或治理决策而过时。

该机制将节点的收益与网络的长期稳定和发展相绑定，防止短期投机行为，确保可靠节点获得可预期的收入流。

节点可查询：

- 待释放代币的总金额
```
http://<ip>:<public_http_port>/chain-api/productscience/inference/streamvesting/total_vesting/<Host_address>
```
其释放计划的详细分解（未来解锁的时间表数组）。
```
http://<ip>:<public_http_port>/chain-api/productscience/inference/streamvesting/vesting_schedule/<Host_address>
```

## 奖励领取流程
Gonka 链仅在第 N+1 轮次中发放在第 N 轮次所获得的奖励。奖励无法提前或延后领取。

主机的 API 节点会在第 N+1 轮次期间，自动使用在第 N 轮次开始时记录签名的种子（seed）执行奖励领取操作。若领取失败，该节点将每 30 分钟重试一次。

然而，领取奖励要求节点在此有限的时间窗口内保持在线，并成功通过所有验证步骤，直至奖励被成功领取。如果无法完成验证，链将无法最终确认奖励，导致奖励无法被领取。

!!! note "关键运行提示"

    奖励领取操作**仅由主机的 API 节点**执行。
    
    如果在第 N+1 轮次期间，API 节点被停止、重启或处于不可用状态，
    即使满足以下所有条件，第 N 轮次的奖励仍可能永久丢失：
   
 
    - 已成功完成计算证明（Proof of Compute, PoC）
 
    - 正确响应了推理请求
 
    - ML 节点始终保持在线
    
    **仅靠 ML 节点无法领取奖励**。为确保奖励被安全领取，API 节点必须在整个过程中保持在线并同步状态，直至前一轮次的奖励被确认已成功领取。

在领取奖励时，链需要验证主机是否已完成该轮次的所有必要工作。同时，链也允许主机在奖励领取窗口期内提交逾期的推理验证。这些验证要求主机保留所有推理输出及其相关元数据。

## 种子承诺与验证
种子（seed）是由每个网络节点在每轮次开始前独立生成的一个保密值。在轮次开始前，网络节点需通过向链提交该种子的数字签名，完成对种子的承诺。  
在轮次结束时，为了获得奖励，节点必须公开其原始种子值。链随后会验证所公开的种子是否与其先前记录的签名相匹配。

示例结构：
```
{
  "seed": 5839402176541298043, // the seed itself, revealed during the claim 
  "epoch_index": 101,
  "signature": "a5910d9b4156e403494a0ba148e6fb67e101ab2a7b7cf54eb15fa5bb5a46a2fdb1435a14954221008d294987075b75d247f26fa7d273720478f0ffb3189132f6", // the seed signature, committed beforehand
  "claimed": false
}
```

该机制确保每个主机（Host）真正以随机方式选择推理请求进行验证，且未对选择过程进行操纵。同时，其他主机无法推断该节点如何执行其随机选择，从而防止对抗性适应行为。

## 奖励未申领的原因

尽管申领流程是自动的，但并非无条件执行。

在第 N+1 轮期间，若出现以下情况，奖励可能未被申领：

- API 节点离线或正在重启
- 节点无法提供正确的种子（seed）（终端性故障）
- 所需的推理-验证数据无法验证（终端性故障）
- 网络不稳定导致最终确认失败（可重试故障）

区块链不会在申领窗口期后保留推理数据，也无法验证逾期的申领请求。  
此前各轮未申领的奖励将被永久销毁。

## 奖励申领失败情况矩阵

| 失败条件 | 发生时机 | 结果 | 自动重试 | 是否支持手动申领 | 最终状态 |
| --- | --- | --- | --- | --- | --- |
| 短时网络不稳定 | 第 N+1 轮期间 | 申领尝试失败，但奖励仍处于待处理状态 | 是（每 30 分钟一次） | 是（可选） | 在第 N+1 轮结束前可恢复 |
| API 节点短时停机 / 重启 | 第 N+1 轮期间 | 申领延迟 | 是（节点恢复后） | 是（可选） | 在第 N+1 轮结束前可恢复 |
| 节点在线但链上最终确认失败 | 第 N+1 轮期间 | 申领未完成最终确认 | 是 | 是 | 在第 N+1 轮结束前可恢复 |
| 种子错误或缺失 | 第 N+1 轮期间 | 申领被拒绝 | 否 | 否 | 若种子可恢复，则在 N+1 轮内可恢复 |
| 本地种子丢失 | 第 N+1 轮期间 | 无法授权申领 | 否 | 否 | 永久不可恢复 |
| 推理-验证数据缺失或损坏 | 第 N+1 轮期间 | 验证失败 | 否 | 否 | 永久不可恢复 |
| 过多无效或遗漏的推理请求 | 第 N 轮期间 | 主机被取消资格 | 不适用 | 否 | 第 N 轮奖励被没收 |
| 随机 PoC（验证证明）失败 | 第 N 轮期间 | 主机被惩罚并被禁用 | 不适用 | 否 | 第 N 轮奖励被没收 |
| API 节点在申领完成前关闭 | 第 N+1 轮期间 | 无法完成申领 | 是（若节点及时恢复） | 是（若仍在 N+1 轮内） | 在第 N+1 轮结束前可恢复 |
| 申领窗口已过期（第 N+1 轮结束） | 第 N+1 轮之后 | 不再允许申领 | 否 | 否 | 永久销毁 |

## 手动申领奖励

手动申领功能适用于以下操作人员：

- 正在停用或退役节点
- 在修复了导致第 N+1 轮期间自动申领失败的问题后，需要主动触发申领
- 希望强制立即发起申领尝试，而不愿等待自动重试周期

!!! note "运维说明"
    在第 N+1 轮期间发生短时停机**不需要**手动申领。  
    系统会在整个第 N+1 轮期间**每约 30 分钟自动重试**申领。  
    仅当节点无法及时恢复服务，或操作员希望比下一次自动重试更早发起申领时，才需要手动操作。

    **手动申领无法恢复已过期轮次的奖励**。

    若 API 节点在第 N+1 轮期间处于离线状态且申领窗口已关闭，则第 N 轮的奖励将永久不可恢复。

    手动申领**仅适用于上一个轮次**，且**前提是本地仍保存有正确的种子（seed）**。
```
curl -X POST http://localhost:9200/admin/v1/claim-reward/recover \
    -H "Content-Type: application/json" \
    -d '{"force_claim": true, "epoch_index": 106}'
```
此端点仅在有效的申领周期（N+1）内有效。申领窗口关闭后，过期的奖励将无法再被领取。

## 申领验证流程

申领奖励依赖于节点对推理结果的验证参与。为了确保随机性，每个主机（Host）需在周期开始时提交其将用于选择待验证推理任务的随机种子（random seed）的签名。

在执行申领操作时，节点需提供该随机种子，而该种子的签名已在周期开始时被记录在链上。

链会验证用于本次验证的所有推理抽样事件，是否确实由该特定种子生成。

为了完成此验证，系统需要访问所有推理数据。由于存储成本较高，这些数据仅保留一个周期。

您可通过以下 API 查询推理性能相关信息：
```
GET /chain-api/productscience/inference/inference/epoch_performance_summary/<epochId>/<HostAddress>
```
示例
```
export NODE_URL=http://46.4.101.189:8000/
curl $NODE_URL/chain-api/productscience/inference/inference/epoch_performance_summary/84/gonka1q0a3u8s2azydlu2s2902qxwfz8d7mkze4lf45f | jq
```

返回指定 epoch 和 Host 的性能摘要。

## 奖励可能被没收的条件
在以下情况下，Host 可能丧失获得奖励的资格：

**
1. 在奖励申领完成前关闭节点**

如果 Host 的 API 节点在奖励成功申领之前被停止、重启或变得不可用，区块链将无法完成申领流程。

**
2. 过多未完成的推理请求**

在整个 epoch 期间，系统会持续评估推理的正确性。系统跟踪两种独立的故障模式：

- 过多无效推理（Invalid Inferences）
- 过多未完成推理（Missed Inferences，未完成率超过 10%，在样本量较小时会进行统计误差校正；详见代码实现）

这两个指标在整个 epoch 内进行采样。如果任一指标在任何时刻超过了协议定义的故障阈值，Host 将立即面临以下后果：

- 从活跃 Host 列表中移除
- 抵押保证金被罚没（Collateral slashing）
- 没收当前 epoch 的所有奖励

该 epoch 内 Host 所完成的工作将不会获得任何补偿；所有已贡献的推理请求均视为无偿。具体参数由协议层定义，并可能通过链上治理机制更新。Host 应始终以当前链上实现为准，以获取权威行为规范。

**
3. 提前关闭或管理不当**

在证明（PoC）完成或奖励申领最终确认前过早关闭节点，将导致奖励丢失。  
这些规则由协议层强制执行，不可逆转。

**
4. 确认阶段（随机）PoC 失败**

未能通过确认阶段的随机 PoC 将对 Host 的收益产生直接且重大影响。由于确认 PoC 的存在是为了确保 Host 持续保有其声明的计算能力，失败将被视为该 Host 不再提供其声明 PoC 权重所依赖的算力。因此，协议将实施严格的经济惩罚措施：

- 没收当前 epoch 的所有奖励
- 奖励通过按比例重新分配，隐式转移给其余符合条件的 Host
- 抵押保证金被罚没
- 之前保留的 PoC 权重在奖励计算中作废
